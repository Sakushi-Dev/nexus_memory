# 0.4.0 — 2026-06-20

A diary (Layer V) rework: the diary is now keyed by **session** instead of by
**calendar day**, and the **current** session's entry is **always injected** into
the context (even before it is finalized) so the conversation persists across
session boundaries. The persistent layer changes from a frozen section ring to a
**single growing summary** that is extended every `sessions_per_summary` sessions.
**Breaking for the diary layer only:** the old `diary_days` / `persistent_sections`
tables are replaced by `diary_sessions` / `persistent_summary`; the old tables are
not migrated, so an existing diary host restarts its diary history. **Non-diary
hosts are unaffected** — with the diary off (the default) behavior is byte-identical
to legacy: no new tables, no provider output.

## Added

- **`DiaryConfig.sessions_per_summary`** (default `6`) — finalized session diaries
  folded into the single persistent summary per fold/extension. Replaces the old
  `section_size`.
- **`DiaryConfig.inject_sessions`** (default `1`, **range `0-6`**) — the number of
  **additional previous** finalized session diaries injected into the context. The
  **current** session diary is always injected on top of these, and the single
  `<persistent_summary>` is always included when present. Replaces the old
  `inject_days`. Default `1` ⇒ "current + last".
- **`DiaryConfig.summary_max_sentences`** (default `300`) — the upper bound of the
  single growing persistent summary, formatted into `SUMMARY_PROMPT` at enqueue
  time; floor 2. The old per-section cap (`max_sections` ring) is gone.

## Changed

- **The diary is session-keyed, not day-keyed.** `diary_days` → `diary_sessions`,
  keyed by the orchestrator's `session_id` (a uuid4 per `NexusMemory` process run)
  with a monotone `seq` (1, 2, 3, …) that orders current/previous and triggers the
  fold. The daily UTC-day rollover (`day > last`) is replaced by a **session
  rollover**: when the first interaction of a new session arrives while the
  previously seen session is still open, that previous session is finalized and its
  final rolling job is enqueued (`force=True`).
- **The current session diary is always injected** — including before it is
  finalized. This is the core change: the live session's narrative is in the prompt
  from the start, so context survives across sessions. The provider emits
  `<diary session="current" seq="…">`, then up to `inject_sessions` previous
  finalized session diaries (newest-last), then exactly one `<persistent_summary>`.
  Diary fragments still carry **no `id="…"` attributes** (the needle invariant
  `<fact id="(\d+)">` is untouched).
- **The persistent layer is one growing summary, not a section ring.**
  `persistent_sections` → `persistent_summary`, a singleton row. At the
  `sessions_per_summary` mark the summary is **created**; at every further mark the
  **same** row is **extended** (no freeze, no ring), capped at
  `summary_max_sentences` sentences. `session_count` tracks how many sessions have
  been folded in.
- **`DAILY_PROMPT` → `SESSION_PROMPT`** — "the diary of **this session**" rather
  than "the day"; the first-person-prose rules and the `2-{max_sentences}` range
  are retained.
- **`SECTION_PROMPT` → `SUMMARY_PROMPT`** — given the prior persistent summary and
  the new session entries, **extend** it into one coherent first-person prose
  summary of up to `{summary_max_sentences}` sentences, preserving the throughline
  and dropping redundancy.
- **Outbox job kinds renamed:** `'daily'` → `'session'` and `'section'` →
  `'summary'`. A `session` job targets the `session_id`; a `summary` job targets
  the singleton `'1'`. The `summarization_jobs` outbox is otherwise structurally
  unchanged.
- **`examples/diary_outbox.py`** updated for the session model (rollover, the
  always-injected current session, and the single growing persistent summary).

## Removed

- **`DiaryConfig.section_size`** (→ `sessions_per_summary`),
  **`DiaryConfig.max_sections`** (the section ring is gone), and
  **`DiaryConfig.inject_days`** (→ `inject_sessions`).

## Migration

- **Diary hosts only.** The old `diary_days` / `persistent_sections` tables are not
  migrated; the new `diary_sessions` / `persistent_summary` tables are created fresh
  (`CREATE TABLE IF NOT EXISTS` with new names), so an existing diary host restarts
  its diary history. Layer V is opt-in and pre-1.0, so this is acceptable.
- **Config rename:** replace `section_size=` with `sessions_per_summary=`,
  `inject_days=` with `inject_sessions=` (now capped at 6), and drop `max_sections=`.
  Add `summary_max_sentences=` to tune the persistent-summary cap.
- **Non-diary hosts:** no action. With the diary off (the default) there is no new
  table and no provider output — behavior is byte-identical to legacy.

## Documentation

- Diary architecture, configuration, use-case, and tuning pages updated for the
  session model: session keying and rollover, the always-injected current session,
  the single growing persistent summary (`sessions_per_summary`,
  `summary_max_sentences`), and the `inject_sessions` knob (range `0-6`).
