# 0.4.2 — 2026-06-20

Reply language is no longer Nexus's concern. The procedural layer used to mine
and store language directives ("Respond in German." / "Respond in English.") from
the user's text and inject them into the assembled context; that is removed.
Choosing the reply language is the host application's job, handled out of the box
in its own system prompt (the chat demo does this with a `/lang` switch and a
language line it pins itself), so the memory module stays language-agnostic.

## Changed

- **The procedural detector no longer mines reply-language directives — and no
  longer has a generic `always/never` catch-all at all.** The explicit
  German/English language patterns were removed, the `"language"` category was
  dropped from the valid set (`tone` / `format` / `persona` / `other` remain),
  and the generic "Standing rule: …" catch-all that fired on any sentence
  containing `immer`/`always`/`nie`/`never` is gone. That catch-all could not
  tell a reply-language wish ("antworte immer auf Deutsch") from a genuine rule
  ("zitiere immer Quellen"), so it leaked language directives through the back
  door (and produced near-duplicate rows). `MockDirectiveDetector` now recognizes
  only the two specific, safe patterns: **tone** (concise) and **persona**
  (address-as / name). A host that wants richer rule mining plugs in its own
  `DirectiveDetector`.
- **The semantic extractor no longer importance-boosts reply-language words.**
  `deutsch` / `englisch` / `sprich` / `sprechen` / `sprichst` were removed from
  the high-value token set, so a language instruction is no longer artificially
  promoted to a high-importance fact (which had also fed it into distillation).
  General standing-rule signals `immer` / `nie` stay — they apply to any
  preference, not just language.

## Migration

- If you relied on Nexus auto-detecting the reply language and injecting a
  "Respond in …" directive, set the language yourself in your system prompt
  instead. A manually added rule with `category="language"` now normalizes to
  `"other"` (it is still stored, just uncategorized). No database migration is
  needed.
- If you relied on the generic `always/never` catch-all to auto-capture standing
  rules, note it is gone: add such rules explicitly (the `pin` action /
  `add_rule`) or supply your own `DirectiveDetector`. This is what removes the
  language leak the catch-all caused.

## Documentation

- Removed the "Respond in German." procedural examples across the docs (memory
  layers, behavioral rules, the I/O contract, the API reference, and the
  assembled-context snippets) and replaced them with tone/persona examples;
  clarified that reply language is the host's responsibility, never a stored
  directive.
- New use-case page: **Multiple independent agents** — each `NexusMemory(db_path=…)`
  is bound to one SQLite file and is fully isolated, so running several independent
  bots is simply one database per bot (own facts, history, directives, and diary;
  no cross-talk). Linked from the docs index and the README use-cases list.
